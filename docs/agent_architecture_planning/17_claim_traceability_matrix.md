This document hardens the previous architecture bundle by mapping important conclusions to concrete local source or documentation evidence. It is intentionally stricter than the earlier synthesis: weak claims are marked as Medium/Low and must not drive implementation without a prototype or eval.

## Review Rules

- **Observed** means the cited path explicitly contains the mechanism.
- **Inferred** means the cited structure strongly implies the behavior, but a runtime trace or deeper code path is still needed.
- **Adopt** means adopt the pattern, not necessarily copy the implementation.
- Exact excerpts are kept short; use the file paths for full context.

## Critical Gaps Found During Review

1. **Several earlier claims were too broad.** For example, "DeepSeek optimization" must be decomposed into parser behavior, prefix-cache preservation, compaction thresholds, reasoning preservation, and eval evidence.
2. **Qwen evidence is weaker than DeepSeek evidence in the local workspace.** The architecture must include Qwen3.6-27B native mode, but implementation should first refresh and pin Qwen3.6 docs, serving parameters, parser behavior, and tool-call traces.
3. **ClaudeCode adaptation is not just prompt design.** The evidence shows model-aware thinking, tool schema stability, prompt/cache breakpoints, output-token caps, input normalization, and provider-specific safeguards.
4. **Permission systems are only as good as their denial tests.** ClaudeCode/OpenCode/ClawCode all expose policy mechanisms, but our product needs a threat-model-backed eval suite before auto-allowing any command or write.
5. **Plugin/hooks are powerful but dangerous.** They should not be in Product Kernel v0 unless the permission/audit model is already hardened.
